WHAT &
WHY

📌
기획 배경
WealthTracker는 사용자의 소비와 저축 흐름을 기록하고, 그 데이터를 바탕으로 예산 관리, 목표 저축, 예정 결제 추적, 주간 소비 피드백까지 연결하는 개인 재무 관리 시스템이다. 문제는 단순히 거래를 저장하는 것이 아니라, 기록이 비교와 계획, 그리고 실제 행동 변화로 이어지도록 시스템 전체 흐름을 설계해야 한다는 점이었다.
🎯
프로젝트 목표
이메일 인증과 JWT 기반 사용자 식별을 바탕으로, 지출·수입·저축 목표·카테고리 예산·예정 결제를 하나의 문맥으로 연결하고, 사용자가 자신의 재무 상태를 숫자뿐 아니라 비교 가능한 집계와 AI 피드백으로 해석할 수 있는 구조를 만드는 것이 목표였다. 특히 기능 나열이 아니라 데이터 저장, 정합성 제어, 집계, 재사용, 운영 흐름까지 자연스럽게 이어지는 서비스 설계에 집중했다.

HOW I
BUILT IT

1
부하 테스트 기반 병목 분석 · DB/SQL 성능 튜닝
배경

/api/expend/graph는 지출 그래프를 그리기 위한 핵심 조회 API였고, 1000명 동시 요청과 100만 건 더미 데이터 기준 부하 테스트에서 평균 4004ms, P95 11000ms 수준까지 응답이 지연되고 있었습니다. Locust, Prometheus, Grafana를 통해 실제 병목 구간을 먼저 확인했습니다.

도전 과제
  • 유저 수 증가 구간에서 Hikari 커넥션 풀 대기 수가 급격히 증가
  • 이번 달 / 저번 달 데이터를 각각 조회하는 중복 쿼리 구조
  • 주차별 DTO를 서비스 계층에서 다시 조립하는 비효율 존재
해결 방안
  • HikariCP 풀 크기를 로컬 코어 수와 DB 최대 연결 수 기준으로 조정 ↗ 내용 자세히 보기
  • UNION ALL 기반 Native Query로 이번 달 / 저번 달 집계를 한 번에 조회
  • expendDate + userId 복합 인덱스 적용으로 조회 경로 단순화 ↗ 내용 자세히 보기
결과
  • 평균 응답 시간: 4004ms → 722.59ms → 642.51ms
  • P95 응답 시간: 11000ms → 2200ms → 2100ms
  • 캐시 적용 전 단계에서 DB 병목과 쿼리 구조를 먼저 정리하는 1차 개선을 완료했습니다.
2
Caffeine 캐시 적용 · 최종 응답 시간 개선
cache 적용 Diagram
배경

쿼리와 커넥션 풀 튜닝 이후에도 /api/expend/graph는 조회 빈도가 높은 API였기 때문에, DB를 매번 타지 않도록 캐시를 적용하는 것이 가장 큰 개선 포인트로 남아 있었습니다.

도전 과제
  • 조회 비율이 높지만 지출 작성·수정·삭제 시 stale data를 방지해야 했음
  • Redis를 추가로 붙이는 것은 당시 상황에서는 과한 선택이었음
해결 방안↗ 내용 자세히 보기
  • Spring Cache + Caffeine 적용
  • JWT token 기준으로 사용자별 캐시 키 분리
  • 작성 / 수정 / 삭제 API에서 @CacheEvict로 해당 사용자 캐시 무효화
  • 10분 TTL 설정으로 불필요한 캐시 자동 정리
결과
  • 평균 응답 시간: 642.51ms → 108.2ms
  • P95 응답 시간: 2100ms → 480ms
  • 최종적으로 최초 대비 평균 4004ms → 108.2ms, P95 11000ms → 480ms를 달성했고, 목표였던 평균 500ms / P95 1000ms도 초과 달성했습니다.
3
MySQL Named Lock 기반 카테고리 동시성 제어
Named Lock Sequence Diagram
배경

사용자가 새로운 지출 카테고리를 직접 추가할 수 있었는데, 여러 사용자가 동시에 같은 이름의 custom category를 등록하면 중복 insert가 발생하는 Race Condition이 존재했습니다.

도전 과제
  • 같은 카테고리명에 대한 동시 insert 경쟁 발생
  • @Transactional만으로는 멀티스레드 환경에서 정합성 보장 불가
  • Redis, Zookeeper 같은 외부 락은 당시 프로젝트 규모 대비 비용이 컸음
해결 방안
  • MySQL GET_LOCK / RELEASE_LOCK 기반 Named Lock 선택
  • 카테고리명을 lock key로 사용해 해당 이름에 대해서만 직렬화
  • lock 획득 후 findByCustomCategoryName → 없으면 save 흐름으로 getOrCreate 구현
결과
  • 동시에 같은 카테고리명을 등록해도 최종 생성 객체 수를 1개로 제어
  • 외부 인프라 추가 없이, 기존 MySQL만으로 동시성 문제를 해결하는 구조를 만들었습니다.
4
Gemini 기반 주간 소비 피드백과 재사용 전략
AI 피드백 Diagram
배경

숫자와 그래프만으로는 사용자가 자신의 소비 패턴을 빠르게 해석하기 어렵다고 판단했습니다. 그래서 이번 주 소비를 자연어로 요약해 주는 피드백 기능을 추가해, 사용자가 데이터를 단순히 조회하는 수준을 넘어 실제로 이해하고 행동으로 연결할 수 있도록 만들고자 했습니다.

도전 과제
  • 카테고리별 주간 지출 데이터를 AI가 해석할 수 있는 입력 형태로 가공해야 했음
  • 같은 주간 데이터에 대해 반복 요청이 들어올 때마다 외부 AI를 다시 호출하면 비용과 응답 시간이 불필요하게 증가할 수 있었음
  • 사용자가 지출을 수정하거나 삭제한 경우, 기존 피드백이 더 이상 최신 데이터와 일치하지 않을 수 있었음
해결 방안
  • 식비, 교통, 쇼핑 등 카테고리별 주간 지출 합계를 기준으로 Gemini용 프롬프트를 직접 구성
  • 같은 주간에 대해 이미 생성된 피드백이 있고, 이후 지출 데이터에 변경이 없다면 기존 피드백을 재사용하도록 설계
  • 최신 지출 생성·수정 시각과 피드백 생성 시각을 비교해, 필요한 경우에만 AI를 다시 호출하도록 처리
  • @Scheduled를 이용해 오래된 피드백을 주기적으로 정리하고, 주간 단위 라이프사이클을 유지
결과
  • 숫자 데이터가 사용자가 이해할 수 있는 자연어 소비 피드백으로 확장됨
  • 외부 AI 연동을 단순 호출이 아니라, 재사용과 갱신 조건까지 포함한 구조로 설계
