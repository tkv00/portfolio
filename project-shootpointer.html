1
OpenCV 협업 기반 하이라이트 생성 파이프라인
Highlight Sequence Diagram
배경

ShootPointer는 농구 영상을 올리면 개인 하이라이트를 생성하고, 그 결과를 게시글·검색·랭킹까지 연결하는 서비스였습니다. 핵심은 영상 분석 자체보다, OpenCV 서버와 백엔드가 역할을 분리해 안전하게 협업하는 구조를 만드는 것이었습니다.

도전 과제
  • 백엔드가 직접 영상 처리하지 않고 외부 OpenCV 서버와 협업해야 했음
  • 업로드 권한, 사용자 식별, 작업 추적을 안전하게 연결해야 했음
  • 하이라이트 생성 진행 상황을 사용자에게 실시간으로 보여줘야 했음
해결 방안
  • 백엔드는 등번호 등록, 회원-등번호 매핑, 하이라이트 결과 저장 API를 담당하고, OpenCV 서버는 개인 식별과 클립 생성을 담당하도록 책임을 분리 ↗ PR #40 ↗ PR #48 ↗ PR #52
  • OpenCV가 생성한 결과를 /api/highlight/upload-result로 수신해 DB에 저장하고, 이후 게시글 작성과 조회 흐름에서 재사용 가능하도록 구성 ↗ PR #52
  • 백엔드가 미디어 처리 자체를 맡지 않고, 외부 처리 서버와 협업하는 구조로 서비스 서버의 책임을 명확히 유지
결과
  • 단순 영상 업로드가 아니라 “업로드 → 처리 → 진행률 확인 → 결과 저장 → 게시글 연결”까지 이어지는 서비스 흐름을 완성
  • 외부 미디어 처리 서버와 백엔드의 책임을 분리한 구조 확보
2
Presigned Upload + SSE 기반 실시간 진행률 처리
업로드 권한 발급부터 결과 조회까지 하나의 jobId로 추적하는 흐름
배경

영상 업로드를 바로 OpenCV 서버로 보내는 구조에서는, 누가 어떤 권한으로 업로드하는지와 작업 ID를 어떻게 추적할지, 그리고 진행률 구독과 결과 조회를 job 단위로 어떻게 연결할지가 중요했습니다.

도전 과제
  • 업로드 권한을 백엔드가 통제하면서도 OpenCV 서버로 직접 업로드되게 해야 했음
  • 업로드 요청과 후속 처리 과정을 하나의 작업 단위로 추적할 식별자가 필요했음
  • 처리 중인 진행률을 사용자에게 실시간으로 전달해야 했음
해결 방안
  • API 서버가 파일 메타데이터를 검증한 뒤 Pre-signed URL과 jobId를 발급하고, Redis에 memberId → jobId를 TTL과 함께 저장해 업로드 요청을 작업 단위로 관리 ↗ PR #202
  • OpenCV 서버는 업로드 후 진행률을 Redis Pub/Sub에 발행하고, API 서버는 이를 구독해 /api/progress/subscribe SSE로 클라이언트에 중계 ↗ PR #202 ↗ PR #245
  • 처리 완료 후에는 같은 jobId 기준으로 방금 생성된 하이라이트만 조회할 수 있도록 /api/highlight/latest 흐름을 분리 ↗ PR #242 ↗ PR #244
결과
  • 업로드 권한, 작업 식별, 진행률 구독, 결과 조회가 하나의 job 단위로 연결됨
  • 단순 파일 업로드가 아니라 추적 가능한 비동기 처리 흐름을 구축
3
좋아요 동시성 제어 · 10,000건 정합성 보장 ↗ 내용 자세히 보기
좋아요 동시성 문제 ↗ 내용 자세히 보기
배경

게시판 기능에서 좋아요 수를 Post 테이블에서 직접 관리하고 있었기 때문에, 다수 사용자가 동시에 요청을 보내면 좋아요 개수 정합성이 깨질 가능성이 있었습니다. 실제로 10,000건 동시 요청 테스트에서 기대값 10,000이 아니라 1,528만 반영되는 문제가 드러났습니다.

도전 과제
  • @Transactional만으로는 멀티스레드 환경 정합성을 보장할 수 없었음
  • 100 / 1,000 / 10,000건 동시 요청 기준으로 다양한 방식의 성능과 정확성을 비교해야 했음
  • 애플리케이션 레벨, DB 레벨, 외부 락 방식을 모두 검토해야 했음
해결 방안
  • Atomic, Optimistic Lock, Pessimistic Lock, Redis Distributed Lock을 동일한 동시 요청 조건에서 비교할 수 있도록 테스트 코드를 구성 ↗ PR #69
  • 좋아요 엔티티 생성/삭제와 likeCnt 변경이 함께 일관성을 유지하도록 비관적 락 중심의 구조로 정리 ↗ PR #72
  • 단순히 가장 빠른 방법이 아니라, 실제 운영에서 정합성을 우선하는 방식으로 최종 전략을 선택
결과
  • Increment Query, AtomicInteger, Optimistic Lock, Pessimistic Lock, Redis 분산락 등을 비교
  • 단순히 빠른 방식이 아니라 실제 운영 환경에서 정합성을 우선하는 방식으로 선택
  • 최종 구현은 DB 비관적 락 중심으로 정리

ElasticSearch 기반 검색 품질 개선
Elastic Search 검색 정확도
배경

기존 검색은 제목·본문에 대한 SQL LIKE 기반 검색 후 최신순으로 정렬하는 구조였습니다. 이 방식은 검색 정확도 중심 정렬이 어렵고, 자동완성이나 검색 랭킹 같은 기능 확장에도 한계가 있었습니다.

도전 과제
  • 제목과 본문 중 어디에 검색어가 있는지에 따라 중요도를 다르게 반영해야 했음
  • 좋아요 수와 최신성까지 함께 고려한 정렬이 필요했음
  • 단순 구현이 아니라 검색 정확도를 정량적으로 평가해야 했음
해결 방안
  • Docker 기반 Elasticsearch + Kibana + Nori 형태소 분석기 구축 ↗ PR #113
  • 제목/본문 가중치, 자동완성, 정렬 기준, NoOffset + Slice 기반 조회를 적용해 검색 결과의 relevance를 개선 ↗ 내용 자세히 보기 ↗ PR #125 ↗ PR #129
  • rank-aware 지표(MRR, MAP, nDCG@10)로 검색 품질 평가 ↗ PR #210
결과
  • nDCG@10 0.632 → 0.941
  • MRR@10 0.642 → 1.0
  • MAP@10 0.476 → 0.776
  • 평균 정확도 55.89% 향상

CI/CD 배포 시간 4.5배 개선 · Docker 최적화↗ 내용 자세히 보기
CI/CD Architecture
배경

ShootPointer는 Jenkins + GitHub Actions + Webhook 기반 CI/CD를 운영 중이었고, Docker Compose 멀티 컨테이너 아키텍처로 배포하고 있었습니다. 그런데 배포마다 최소 5~6분, 최대 8~9분의 다운타임이 발생해 팀 테스트 흐름을 자주 막고 있었습니다.

도전 과제
  • Docker build + deploy 구간이 가장 큰 병목이었음
  • 컨테이너 삭제와 이미지 빌드가 느렸고, Docker가 디스크의 큰 비율을 차지하고 있었음
  • 개발 단계에서 잦은 배포를 감당할 수 있도록 배포 시간을 줄여야 했음
해결 방안
  • Redis 컨테이너 제거 후 서버 외부 프로세스로 분리
  • OpenJDK slim 대신 alpine 계열 JRE 이미지 적용
  • 멀티스테이지 빌드로 builder / running 이미지 분리
  • .dockerignore, BuildKit, Jenkins 최적화 적용
결과
  • Docker Image 용량 6.5GB → 4.9GB (24.51% 절감)
  • Docker Containers 용량 2.1GB → 1.9GB(9.76% 절감)
  • 최대 4.48배 빠른 배포 속도
  • 최소 1.34배 빠른 배포 속도
